       - `createSubtaskSchema`: z.object({ title, description, assigneeAgentId (optional), priority (optional) })
       - `addDependencySchema`: z.object({ dependsOnId: z.string().uuid() })
       - Export from shared validators barrel and shared index

    3. Extend `server/src/services/issues.ts` with new methods on the service:
       - `createSubtask(companyId, parentIssueId, data)`: creates an issue with parentId set, returns the new issue. Validates parent exists and belongs to same company.
       - `listSubtasks(companyId, parentIssueId)`: returns subtasks with their dependency edges, ordered by topological sort. Uses dependency-graph.ts topologicalSort. Include dependsOn and dependedOnBy arrays for each subtask.
       - `addDependency(companyId, issueId, dependsOnId)`: inserts edge into issue_dependencies. First validates both issues exist, share same parent, and belong to company. Then validates no cycle would result (fetch all current edges for the parent, add the new one, call validateNoCycle). Uses a transaction for atomicity.
       - `removeDependency(companyId, issueId, dependsOnId)`: deletes edge from issue_dependencies.
       - `deriveParentStatus(companyId, parentIssueId)`: computes parent status from subtask states:
         - All done/cancelled -> "done"
         - Any blocked or has unmet dependencies -> "blocked"
         - Any in_progress -> "in_progress"
         - Otherwise -> "ready"
       - `updateSubtaskStatus(companyId, issueId, status)`: updates subtask status AND re-derives parent status, updating parent if changed.

    4. Export `topologicalSort`, `validateNoCycle`, `getExecutionWaves` from `server/src/services/index.ts`

       - `POST /companies/:companyId/issues/:issueId/dependencies` -- add dependency (validate with addDependencySchema)
       - `DELETE /companies/:companyId/issues/:issueId/dependencies/:dependsOnId` -- remove dependency
